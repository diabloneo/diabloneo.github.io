Golang源码阅读笔记 -- go build(4)

Mar 4, 2017

runBuild函数

在之前的三篇文章中，我们从runBuild函数开始，阅读了获取package信息的过程，然后又回到了runBuild函数中。现在，我们继续来看runBuild函数剩余的代码。先回顾下runBuild的代码：

func runBuild(cmd *Command, args []string) {
	instrumentInit()
	buildModeInit()
	var b builder
	b.init()

	pkgs := packagesForBuild(args)

  // 剩余部分

现在，我们来看下剩余部分的代码的主要流程：

  1. 创建一个builder对象b，这个对象是用来保存build过程的状态的。
  2. 为要构建的package，生成一个action对象a，用来表示构建过程要执行的动作。
  3. 调用b.do(a)执行这些构建的动作。

下面我们会仔细的来看builder对象和action对象的内容。这里总结一下go build命令的逻辑：获取package信息之后，生成需要执行的动作，并且为这些动作排序，然后执行这些动作。这些动作其实就是编译、打包、连接等编译过程的基础动作。

action对象

action对象位于文件cmd/go/build.go中，表示build的动作序列中的一个动作。action对象本身没有方法，只保存一个动作所需的信息。其中几个主要的成员如下：

  • p *Package 该action要作用到那个package上
  • f func(*builder *action) error 这个是该action要执行的处理
  • deps []*action 该action依赖于哪些action，要等这些action完成后才能执行本action
  • triggers []*action 哪些action依赖本action
  • pending int 还有多少依赖的action没有执行

builder对象

builder对象保存一次build过程的全部状态，负责构建action序列以及执行action序列，该对象位于cmd/go/build.go文件中。使用builder对象主要是使用它的几个方法，下面我们来一一介绍。

(b *builder) init方法

该方法在runBuilde中被调用，负责初始化builder对象，以及创建一个临时工作目录用于保存构建过程的中间结果。该临时目录一般是/tmp/go-build开头的，其值保存在b.work中。

(b *builder) action方法

该方法直接调用(b *builder) action1(mode buildMode, depMode buildMode, p *Package, lookshared bool, forShlib string) *action方法。

action1这个方法会构建一个package对应的action及其依赖的action，来看一下它的逻辑你就会明白。

这里要说明一下参数：

  • lookshared和forShlib可以认为是false和空字符串，这个在常用的构建过程不起作用。
  • mode表示要构建的package的方式，值是modeBuild
  • depMode表示依赖的package的模式，默认是modeBuild，如果从命令行指定了-i选项，则这个值为modeInstall
  • p就是该动作要处理的package

下面来看一下action1的主要逻辑：

  1. 先遍历p的所有imports，也就是p的所有依赖，然后为每个依赖调用action1方法，并将结果添加到a.deps列表中。这里使用递归调用，可以直接生成构建一个package所需要的所有action。
  2. 当依赖处理完后，就会开始生成当前的action
    1. 确定a.objdir，长这样$WORK/goexample/hello/_obj/，这个是一个package编译出来的.o文件的存放位置。
    2. 确定a.objpkg，长这样$WORK/goexample/hello.a，这个是一个package编译出来的.o文件打包后的位置。
    3. 确定a.link，如果要构建的package名称是main，那么该值为true，表示最后需要进行连接，生成可执行文件。
  3. 接下来，根据mode参数确定要action的其他内容。
    1. 如果是modeBuild：
      1. 设置a.f的值为(*builder).build，也就是builder对象的build方法。
      2. 设置a.target = a.objpkg
      3. 如果a.link为true，那么会修改a.target，长这样$WORK/goexample/hello/_obj/exe/a.out

经过上面的处理后，就得到了要构建一个package的action序列，这个序列其实是一棵树，上层节点依赖于下层节点。这里的重点是(*builder)build这个方法，它完成了构建的动作。

(b *builder) build方法

(b *builder) build(a *action) (err error)方法负责编译和打包一个package，编译和打包的动作从它这里发起，但是并不在这里实现。我们来看这个方法的逻辑：

  1. 创建a.objdir这个目录以及a.target的目录部分。
  2. 接下来会做一些和cgo以及swig相关的工作，我们先跳过。
  3. 接下来，如果a.p.coverMode不为空，那么表示这个是需要进行代码覆盖率计算的处理，我们也跳过。
  4. 调用buildToolchain.gc。这里buildToolchain是一个interface的全局变量，因为Golang的构建可以使用Google的编译器，也可以使用gccgo，所以这里就用一个interface来屏蔽具体的工具链的差异。我们在下面再看buildToolchain的实现。这里先看一下这个调用的结果： ofile, out, err := buildToolchain.gc(b, a.p, a.objpkg, obj, len(sfiles) > 0, inc, gofiles)，这个调用会把这个package下的所有go文件编译成为一个目标文件ofile。编译得到的文件可能是.o文件，也可能是.a文件。如果是.a文件，说明已经打包过了。
  5. 接下来，会处理.h, .c, .s等文件，这里也略过不讲。
  6. 如果刚才编译得到的是.o文件，那么会调用buildToolchain.pack方法把刚才的.o文件打包成一个.a文件。
  7. 最后，如果a.link为true，那么会调用buildToolchain.ld来进行连接。

现在，我们大概了解了build方法的逻辑，下面我们先来看下toolchain相关的逻辑，然后再回到builder对象。

gcToolchain

上面我们提到了buildToolchain这个全局变量，它的定义是var buildToolchain toolchain = noToolchiain{}，其中toolchain是一个interface，也在文件cmd/go/build.go文件中定义，主要定义了我们前面提到过的gc等借口。在cmd/go/build.go的init函数中，buildToolchain变量会根据go/build.Default.Compiler的值来设置。go/build这个package我们之前已经接触过不少了，在go build(1)中我们已经知道，buildContext就是指向go/build.Default。go/build.Default.Compiler的值是指向runtime/compiler.go文件中的Compiler常量，值是gc。所以，默认的编译是使用Google的编译器，也就是这里的gc。不过，编译器是可以使用-compiler选项修改的。大部分情况下，我们是使用gc编译器，所以这里只讲解gc编译器的逻辑。

使用gc编译器时，buildToolchain变量会被赋值为gcToolchain{}，所以在go build中使用gc编译器都是通过gcToolchain{}的方法来实现的。这里我们来看两个主要的方法：gc和ld。

(gcToolchain) gc方法

这个方法做的主要工作就是以合适的参数调用go tool compile，也就是Golang的编译器。假设我们现在在${GOPATH}/goexamples/hello/目录下执行go build -v -x -work hello.go，那么就可以看到执行了compile命令：

WORK=/tmp/go-build862488510
command-line-arguments
mkdir -p $WORK/command-line-arguments/_obj/
mkdir -p $WORK/command-line-arguments/_obj/exe/
cd /home/diabloneo/go/src/goexamples/hello
/opt/go/pkg/tool/linux_amd64/compile -o $WORK/command-line-arguments.a -trimpath $WORK -p main -complete -buildid b9c10ab5f09ab93c4d4e4dda286cebc203c4d23e -D _/home/diabloneo/go/src/goexamples/hello -I $WORK -pack ./hello.go
...

最后的这行compile命令的生成和调用，就是在(gcToolchain) gc方法中实现的，可以看到编译得到的结果是 $WORK/command-line-arguments.a这个文件。

(gcToolchain) ld方法

这个方法则是调用go tool link，也就是Golang的连接器（代码的构建过程一般都分为编译和连接两个部分）。接着上面的例子，我们可以看到，go build -v -x -work hello.go的输出中也执行了link命令：

/opt/go/pkg/tool/linux_amd64/link -o $WORK/command-line-arguments/_obj/exe/a.out -L $WORK -extld=gcc -buildmode=exe -buildid=b9c10ab5f09ab93c4d4e4dda286cebc203c4d23e $WORK/command-line-arguments.a
mv $WORK/command-line-arguments/_obj/exe/a.out hello

link程序把编译生成的command-line-arguments.a和标准库连接起来，得到了$WORK/command-line-arguments/_obj/exe/a.out这个文件，然后这个文件被重命名为hello。
